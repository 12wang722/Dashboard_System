这是一个测试编排 skill，不直接替代接口测试、UI 测试、缺陷报告或测试用例设计 skill，而是负责识别输入、分配任务、组织执行顺序、调用前置 skill 并汇总最终测试报告。

## 调度目标

完成任务时，默认同时追求以下结果：

1. 判断输入完整度。
- 识别用户提供了哪些材料：需求文档、接口文档、测试地址、账号、环境、截图、补充约束。

2. 选择应调用的子 skill。
- 根据任务目标和输入材料，决定是否调用：
  - `testcase-xmind-generator`
  - `api-doc-to-pytest`
  - `web-ui-playwright-mcp`
  - `bug-report-generator`

3. 形成测试闭环。
- 从需求理解、测试设计、自动化生成/执行、缺陷整理，最终汇总成测试报告。

4. 控制执行范围。
- 优先覆盖用户明确关心的系统范围，不无边界扩张。

## 强制工作流

必须按顺序执行以下步骤。

1. 先识别用户意图与输入材料。
- 若用户说“帮我测试下这个系统”“做一轮测试并给报告”，默认进入系统测试编排模式。
- 先整理已提供材料：需求文档、接口文档、测试地址、账号信息、测试范围、时间要求、输出格式。
- 先输出缺口清单，明确还缺哪些关键输入。

2. 判断本轮需要启用哪些子 skill。
- 若提供了需求文档或需求描述，需要测试点设计或测试范围拆解，调用 `testcase-xmind-generator`。
- 若提供了接口文档、OpenAPI、Postman、curl 或希望生成并执行接口自动化，调用 `api-doc-to-pytest`。
- 若提供了 Web 地址、页面流程、测试环境或希望验证页面交互，调用 `web-ui-playwright-mcp`。
- 若在执行中发现失败现象、异常日志、截图证据，需要整理缺陷，调用 `bug-report-generator`。
- 若某类材料缺失，则不要强行调度对应 skill。

3. 先设计总测试计划，再进入执行。
- 至少拆出：测试范围、测试维度、执行顺序、输入材料映射、输出物清单。
- 默认顺序优先为：
  1. 需求/范围理解
  2. 测试点或用例设计
  3. 接口测试生成/执行
  4. UI 自动化验证
  5. 缺陷整理
  6. 测试报告汇总
- 若用户明确要求只测接口或只测 UI，则缩减流程。

4. 调度子 skill 时保持边界清晰。
- 每个子 skill 只负责自身领域产出，不在编排层重复创造其内部规则。
- 编排层负责：
  - 决定是否调用
  - 传递输入上下文
  - 接收结果
  - 汇总风险、结论和报告

5. 汇总测试结果。
- 至少汇总：
  - 测试范围
  - 已执行项
  - 未执行项与原因
  - 接口测试结论
  - UI 测试结论
  - 缺陷清单/风险项
  - 总体测试结论
- 若存在自动化脚本、截图、XMind、缺陷报告路径，统一列出。

6. 输出最终测试报告。
- 最终报告默认应适合直接发给项目相关方阅读。
- 事实、风险、阻塞和建议分开表达，避免混写。

## 子 Skill 调度规则

### 1. `testcase-xmind-generator`

在以下场景优先调用：

- 用户给了需求文档，希望先梳理测试点
- 用户没有明确测试范围，需要先拆解模块与场景
- 用户希望先产出测试用例或 XMind 再测

典型输出：

- 测试范围拆解
- 测试点 / 用例
- XMind 或结构化 case-tree

### 2. `api-doc-to-pytest`

在以下场景优先调用：

- 用户给了接口文档
- 用户要求生成并执行接口自动化
- 系统测试中需要覆盖核心接口链路

典型输出：

- pytest 接口自动化代码或执行入口
- 接口覆盖摘要
- 接口风险或失败点

### 3. `web-ui-playwright-mcp`

在以下场景优先调用：

- 用户给了测试地址或页面入口
- 用户要求验证 UI 主流程、页面交互、响应式或登录流程
- 系统测试需要做端到端页面验证

典型输出：

- MCP 执行结论
- 截图与交互验证结果
- Playwright 脚本或复现脚本

### 4. `bug-report-generator`

在以下场景优先调用：

- 接口测试或 UI 测试发现异常
- 有截图、日志、录屏、报错信息需要整理成缺陷
- 最终报告需要带结构化缺陷单或缺陷摘要

典型输出：

- 缺陷报告正文
- 标题候选
- 缺陷证据清单

## 调度原则

- 不因为 skill 存在就全部调用；必须按输入材料和目标选择。
- 优先做能产生有效结论的测试，不在关键信息缺失时假装完成。
- 编排层默认先追求“范围清晰 + 关键链路可验证 + 风险可说明”。
- 若某一子 skill 无法执行，必须在总报告中明确影响范围。

## 默认最终输出

完成任务时，默认输出应包含：

- 测试背景与输入材料摘要
- 本轮调度了哪些 skill，以及各自产出
- 测试范围与执行范围
- 已完成测试项
- 未完成测试项与原因
- 缺陷/风险摘要
- 总体测试结论
- 附件或产物路径清单
