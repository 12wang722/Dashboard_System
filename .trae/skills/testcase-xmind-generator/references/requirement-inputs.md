# 需求输入类型

用于需求抽取的输入类型支持如下：

- 文本文件：`.txt`、`.md`、`.csv`、`.log`、`.yaml`、`.yml`
- 结构化文本：`.json`
- 需求文档：`.docx`
- 需求图片：`.png`、`.jpg`、`.jpeg`、`.bmp`、`.webp`、`.tif`、`.tiff`

输入范围约束：

- 仅处理用户在当前任务中显式提供的输入。
- 用户未提供的需求文档、历史生成的 `output/requirements/*` 文件、仓库中的其他需求资料，不得自动读取。
- 对图片输入，只能基于用户当前提供的图片和当前对话说明做 OCR/理解；不得自行补充读取其他需求文档。

图片 OCR 后端优先级：

1. `pytesseract`（依赖 `pillow` 和本地 Tesseract 可执行程序）
2. `rapidocr_onnxruntime`

若以上两种 OCR 后端都不可用，`extract_requirements.py` 会返回明确的依赖错误信息。

图片输入处理要求：

- 图片输入不能只做 OCR 文本提取，必须继续做视觉理解。
- 视觉理解至少应补充：
  - 页面区域与模块含义
  - 字段、按钮、标签之间的关系
  - 圈选、高亮、箭头、红字批注表达的变更意图
  - 仅图片中可见、OCR 不稳定但对需求有影响的信息
- 视觉理解时，必须按“界面语义”而不是“零散文字”组织结论，至少明确：
  - 当前页面或弹窗的用途是什么
  - 页面分成了哪些区域，各区域承担什么功能
  - 关键控件分别是什么，它们之间是什么关系
  - 用户从哪里进入、做什么操作、系统会发生什么变化
  - 当前图中哪些内容是原有能力，哪些内容是新增、修改、删除或规则调整
- 若图片中存在批注、箭头、高亮、红字说明、圈选区域、删除线、替换说明，必须优先围绕这些标注理解需求，不能只复述底图文字。
- 对这类带标注图片，视觉理解输出必须至少回答：
  - 新增了什么
  - 修改了什么
  - 删除了什么
  - 哪条规则被调整
  - 哪个范围或显示方式发生变化
- 若图片只展示局部弹窗、表单片段或配置区域，必须结合局部上下文说明其所处场景，避免把局部控件误解为完整页面需求。
- 若 OCR 结果与视觉理解冲突，应以界面结构、标注关系和视觉上下文为主，并在中间需求 Markdown 的“待确认问题”中记录冲突点。
- 完成 OCR 与视觉理解后，必须先生成一份 `normalized-requirements.md` 再进入测试用例设计。

提升 OCR 质量建议：

- 优先使用 1080p 及以上清晰截图
- 文字尽量保持水平
- 避免明显模糊或高压缩失真
- OCR 前先裁剪无关 UI 区域
